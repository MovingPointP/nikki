# プロジェクトルール

## ページ・コンポーネント作成時の必須手順

**新しいページやコンポーネントを作成する前に、必ず `src/theme.ts` を読むこと。**

テーマで定義されている色・フォント・コンポーネントスタイルを把握し、それに合わせた実装を行う。

### テーマの概要（`src/theme.ts`）

| 項目 | 値 |
|---|---|
| primary.main | `#CA7842` |
| primary.dark | `#4B352A` |
| secondary.main | `#B2CD9C` |
| background.default | `#F0F2BD` |
| background.paper | `#FAFAF3` |
| text.primary | `#4B352A` |
| text.secondary | `#7A5C47` |
| UIフォント | Noto Sans JP |
| コンテンツフォント | Playfair Display / Noto Serif JP |

ハードコードした色指定は避け、`theme.palette.*` を参照する形で実装すること。

## Zustand の使い方

コンポーネントでストアを参照する際は、以下のルールに従うこと。

| 用途 | 書き方 |
|---|---|
| JSX の表示に使う値 | `useStore((s) => s.xxx)` でセレクタ購読 |
| アクション（関数） | `useStore.getState().xxx` で取得 |
| エフェクト内で一度だけ読む値 | `useStore.getState().xxx` で取得 |

セレクタなしの `useStore()` は使わない。ストア全体を購読することになり、関係のないフィールドの変化でも再レンダリングが発生する。

```ts
// NG
const { savePath, setSavePath } = useSettingsStore();

// OK
const savePath = useSettingsStore((s) => s.savePath);
const { setSavePath } = useSettingsStore.getState();
```

## コメントの扱い

既存のコメントは勝手に削除しないこと。リファクタリングや変数名変更を行う際も、コメントを維持したまま内容を更新する。

コードを変更した際、その周辺にあるコメントが内容と合わなくなっていないか必ず確認し、合わなくなっている場合は合わせて更新すること。

---

## コメントのスタイル

ストアやユーティリティなどのファイルを新規作成・編集する際は、以下のスタイルでコメントを付けること。

### セクション区切り

ファイルを論理的なブロックに分割する。

```ts
// ────────────────────────────────────────────
// 定数
// ────────────────────────────────────────────
```

使うセクション名の例：`定数` / `ユーティリティ` / `型定義` / `ストア`

### 型定義のフィールド説明

各フィールドの意味・用途を1行で書く。

```ts
// 未保存の変更がある場合 true
isDirty: boolean;
```

### アクションヘッダー

アクション（関数）の先頭に見出しコメントを付ける。

```ts
// ── 保存パスの更新 ────────────────────────
setSavePath: async (path: string) => {
```

### 非自明な理由コメント

なぜそのコードが必要かが自明でない箇所にのみ補足を書く。

```ts
// autoSave: false のため明示的に save() を呼ぶ必要がある
await store.save();
```
